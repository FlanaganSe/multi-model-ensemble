# Markdown Viewer Research

## 1. Current State of the Brief Viewer

**Location:** `src/features/artifact-viewer/ArtifactViewer.tsx:138-161`

The `BriefView` sub-component renders markdown as plain text inside a `<pre>` tag with `whiteSpace: pre-wrap`. There is no parsing, no heading hierarchy, no table rendering, no syntax highlighting, and no interactive features. The brief string arrives via `getBrief(sessionId)` which calls `invoke<string>("get_brief", { sessionId })` (`src/lib/api.ts:54-56`).

**Data flow:**

```
Rust: synthesis/brief.rs → render_brief() → String
  ↓  tauri command: get_brief
src/lib/api.ts: getBrief() → invoke<string>("get_brief")
  ↓  useState<string | null>
ArtifactViewer.tsx: brief state → BriefView({ brief })
  ↓  currently
<pre>{brief}</pre>   ← plain text, no rendering
```

The `render_brief()` function (`src-tauri/src/synthesis/brief.rs:5-152`) produces well-structured markdown that includes:
- `# Research Brief` heading
- `**Strategy:** ... | **Sources:** ...` bold inline text
- `> **Note:** ...` blockquotes for warnings
- `---` horizontal rules
- `## Key Themes`, `### {label}` heading hierarchy (h2/h3)
- `- item *(source)*` bullet lists with italics
- `**Disagreements:**` bold section headers
- `| Provider | Perspective | Status |` GFM tables with alignment rows
- `*Generated by ...*` italic footer

All markdown is internally generated — no user-supplied content enters this pipeline from web sources.

**Styling approach:** The entire frontend uses inline `style={{}}` objects throughout. There is no CSS file, no Tailwind, no PostCSS config. (`src/main.tsx`, `src/App.tsx`, `src/features/artifact-viewer/ArtifactViewer.tsx` — all inline styles).

## 2. Constraints

**Can't change:**

1. **CSP is restrictive.** `tauri.conf.json:27` sets `"csp": "default-src 'self'; connect-src ipc: http://ipc.localhost"`. No `style-src 'unsafe-inline'` is present. This means:
   - Injected `<style>` tags from CSS-in-JS or library stylesheets will be blocked unless the CSP is extended.
   - `dangerouslySetInnerHTML` for HTML content is not blocked at the CSP level (it's a React API, not a browser injection), but any `<script>` content would be blocked.
   - highlight.js CSS themes, Tailwind's injected styles, or any `<style>` tag inserted at runtime will fail unless `style-src 'unsafe-inline' 'self'` is added to the CSP.
   - **Fix required:** CSP must be updated to add `style-src 'unsafe-inline' 'self'` to support any library that injects a `<style>` tag.

2. **No Tailwind yet.** Tailwind is not configured. Adding `@tailwindcss/typography` requires adding Tailwind + PostCSS first, which is a non-trivial dependency chain (Tailwind v4 uses a different config than v3).

3. **ESM-only ecosystem.** `package.json` uses `"type": "module"`. `tsconfig.json` uses `"moduleResolution": "bundler"`. Libraries must be ESM-compatible (react-markdown v10 is ESM-only — this matches).

4. **TypeScript strictness.** `tsconfig.json:15-19` enables `strict`, `noUnusedLocals`, `noUnusedParameters`, `noUncheckedIndexedAccess`. Any library without good type definitions will require `// @ts-ignore` or declaration patches.

5. **Biome linting.** Biome replaces ESLint/Prettier. Any new code must pass `biome check`. Biome enforces consistent formatting and some rules that conflict with common patterns (e.g., no `dangerouslySetInnerHTML` without explicit review).

6. **No router.** The app uses a custom `View` discriminated union in `App.tsx:7`. No React Router. No view-level code splitting in place, though Vite supports dynamic `import()`.

7. **Vite build target:** `safari13` on macOS (`vite.config.ts:27`). ES2021 TypeScript target. Safari 13 does not support all modern JS features — this rules out any library that uses syntax newer than ES2021 without transpilation (most major libraries are fine).

## 3. Options

### Option A: react-markdown v10 + remark-gfm + rehype-highlight (Recommended path)

**What it is:** react-markdown renders markdown as a React element tree (no `dangerouslySetInnerHTML`, no raw HTML injection). Plugins add GFM table support (`remark-gfm`) and code syntax highlighting (`rehype-highlight` using highlight.js under the hood).

**Dependency chain:**
- `react-markdown` — ~52.6 kB minified (the entire remark/rehype pipeline is included). ESM-only.
- `remark-gfm` — adds GFM: tables, strikethrough, tasklists, autolinks. Small (~5 kB).
- `rehype-highlight` — syntax highlighting via highlight.js. highlight.js itself is ~40 kB minified for the common language bundle; individual language packs can be imported selectively.

**Total estimated addition:** ~100-130 kB minified before gzip (Tauri desktop — bundle size matters less than web, but still relevant). Gzipped: ~35-45 kB.

**CSP impact:** react-markdown does NOT inject `<style>` tags — it produces React elements. However, highlight.js CSS themes ARE a separate CSS import (e.g., `import 'highlight.js/styles/github-dark.css'`). Vite will bundle this as a static stylesheet loaded via `<link>`, which is allowed under `default-src 'self'`. No CSP change required for this option if styles are imported statically.

**UX features enabled:**
- Proper `<h1>`-`<h3>` headings with hierarchy
- GFM `| table |` rendering as `<table>`
- Code blocks with language-aware syntax highlighting
- `**bold**`, `*italic*`, `> blockquote`, `---` horizontal rules — all rendered as semantic HTML
- Custom `components` prop enables: copy-to-clipboard button on `<pre>/<code>`, collapsible `<h2>` sections, custom table styling

**Trade-offs:**
- Pro: No XSS risk (virtual DOM, never calls innerHTML for markdown content)
- Pro: Fully composable — custom components for every element
- Pro: Well-maintained (566+ commits, actively used), strong TypeScript types
- Pro: ESM-only matches project's module type
- Con: Larger bundle than marked (~52 kB vs ~8 kB for marked alone)
- Con: rehype-highlight brings in highlight.js — can be tree-shaken by importing only needed languages
- Con: The plugin API (remark/rehype) has a learning curve

---

### Option B: marked + DOMPurify + dangerouslySetInnerHTML

**What it is:** `marked` parses markdown to an HTML string; `DOMPurify` sanitizes that string; it is injected via React's `dangerouslySetInnerHTML`.

**Dependency chain:**
- `marked` — ~8 kB minified+gzipped. Supports GFM tables natively.
- `dompurify` — ~6 kB minified+gzipped. Strips unsafe HTML tags/attributes.

**Total estimated addition:** ~14 kB gzipped — significantly smaller.

**CSP impact:** `dangerouslySetInnerHTML` is a React API call that writes an HTML string to the DOM. This is not blocked by Tauri's `script-src` CSP on its own. However, if marked's output contains `<style>` tags and the CSP doesn't allow `'unsafe-inline'` for style-src, those would be stripped by DOMPurify anyway. The actual CSP concern is if the sanitized HTML includes inline style attributes — those are retained by DOMPurify by default but would be blocked by a strict `style-src` without `'unsafe-inline'`. In practice, our Rust-generated markdown contains no inline style attributes, so this is not a concern.

**UX features enabled:**
- Headings, tables, bold, italic, blockquotes — all rendered as HTML
- Code blocks: raw `<pre><code>` without highlighting (would need additional JS to walk DOM and apply highlight.js after render)
- No React component tree control — can't inject custom React components into rendered output
- Copy button for code blocks requires a DOM query approach (useEffect + querySelectorAll), not the clean custom-component approach

**Trade-offs:**
- Pro: Much smaller bundle
- Pro: Simpler mental model
- Con: No React component tree — custom behaviors require imperative DOM manipulation
- Con: DOMPurify strips certain patterns that are safe — requires careful configuration
- Con: Less accessible (plain HTML, no ARIA injection points)
- Con: The "no XSS" claim depends entirely on DOMPurify being configured correctly — security surface is larger
- Con: Our markdown is trusted (from our own Rust backend) but the `dangerouslySetInnerHTML` pattern sets a precedent that could be misused later

---

### Option C: markdown-to-jsx

**What it is:** A lighter-weight alternative to react-markdown that converts markdown directly to JSX elements. Smaller dependency tree than the remark/rehype stack.

**Dependency chain:**
- `markdown-to-jsx` — ~11 kB minified (no remark/rehype, self-contained parser)

**CSP impact:** Same as react-markdown — produces React elements, no `<style>` injection.

**GFM table support:** Supported via the `overrides` option and built-in table parsing.

**Trade-offs:**
- Pro: Much smaller bundle (~11 kB vs ~52 kB for react-markdown + plugins)
- Pro: Same security model as react-markdown (virtual DOM)
- Pro: Simpler API for basic use cases
- Con: Plugin ecosystem is limited — no remark/rehype plugins
- Con: Less customizable for advanced use cases (ToC generation, collapsible sections)
- Con: Fewer weekly downloads and a smaller community than react-markdown
- Con: TypeScript types are less robust

## 4. Recommendation

**Use Option A: react-markdown v10 + remark-gfm + rehype-highlight.**

**Rationale:**

1. **This is a desktop app.** Bundle size concerns that dominate web app decisions (LCP, Time-to-Interactive) are largely irrelevant. The app is distributed as a native binary; an extra 100 kB is negligible.

2. **The brief content demands good rendering.** `render_brief()` generates GFM tables (`| Provider | Perspective | Status |`), nested lists with italic source citations, blockquotes, and multiple heading levels. This is not simple paragraph text.

3. **Custom components unlock the UX features.** The `components` prop in react-markdown is the cleanest way to inject copy buttons, collapsible sections, and styled tables. Option B (marked) requires imperative DOM walking. Option C (markdown-to-jsx) has a weaker plugin story.

4. **No CSP change needed for the core path.** Static CSS imports from highlight.js are handled as `<link>` tags by Vite — no `'unsafe-inline'` needed. React element tree approach avoids all HTML injection concerns.

5. **The remark/rehype ecosystem is the industry standard** for programmatic markdown processing. The project may eventually want to add custom remark plugins (e.g., to annotate source citations, linkify provider names) — this investment pays forward.

**Immediate package additions:**
```
pnpm add react-markdown remark-gfm rehype-highlight
```
(`highlight.js` is a peer dependency of `rehype-highlight` and will be installed automatically.)

## 5. UX Enhancement Opportunities

Prioritized by implementation complexity (lowest first):

### Tier 1: Immediate (< 1 day each)

**Syntax highlighting (code blocks)**
- Add `rehype-highlight` with a dark theme matching the app's dark palette
- Import `highlight.js/styles/github-dark.css` (or `vs2015.css` for darker background)
- All ```` ```lang ``` ```` fenced code blocks get color-coded automatically
- Implementation: pass `rehypePlugins={[rehypeHighlight]}` to `<ReactMarkdown>`

**Copy-to-clipboard for code blocks**
- Override the `pre` component in react-markdown's `components` prop
- `navigator.clipboard.writeText(codeText)` — no external dependency needed
- Show a "Copied!" flash state (simple `useState` + `setTimeout`)
- Fully contained in the custom component, zero new dependencies

**Prose typography (without Tailwind)**
- A single ~50-line CSS file (e.g., `src/features/artifact-viewer/brief-prose.css`) can establish:
  - Heading font sizes with proper hierarchy
  - List indentation and spacing
  - Blockquote left-border styling
  - Table `border-collapse` with striped rows
  - `max-width: 80ch` for readability
- This avoids the entire Tailwind setup cost while achieving comparable results
- The app's existing dark color scheme (`#e0e0e0` text, `#111` background) is already established

**Loading skeleton**
- Replace the current plain "Loading session artifacts..." text (`ArtifactViewer.tsx:65`) with animated pulse skeleton bars
- Pure CSS animation, zero dependencies
- Better perceived performance during `getBrief()` invocation

### Tier 2: Medium complexity (1-2 days each)

**Table of Contents (auto-generated)**
- Custom `h2`/`h3` component that collects headings into a sidebar list
- Requires a `useRef` or context to accumulate headings during render
- Smooth scroll to anchors via `element.scrollIntoView({ behavior: 'smooth' })`
- Useful for long comprehensive-strategy briefs with many themes

**Collapsible theme sections**
- Override `h2` with a custom component that renders an expandable `<details>/<summary>` or a state-controlled div
- Default state: first 2-3 sections expanded, rest collapsed
- Especially valuable when `render_brief()` produces 5+ `## Key Themes` sections

**Dark/light theme toggle**
- The app is currently hardcoded dark
- Adding a CSS variable root (`--bg`, `--text`, etc.) and a toggle button enables theming
- highlight.js themes can be swapped by changing the import or toggling a class on the root element

### Tier 3: Higher complexity (3+ days)

**Print/Export to PDF**
- Tauri v2 provides `window.__TAURI__.webview.print()` via the webview API
- Alternatively, `@tauri-apps/plugin-dialog` + `@tauri-apps/plugin-fs` could allow saving the rendered HTML to a file
- Requires careful CSS print media query styling to produce clean output

**Table responsive wrapping**
- GFM tables in the brief can have 3 columns; wrapping on narrow windows needs CSS `overflow-x: auto` container

## 6. CSS/Styling Approach

**Recommended: A single static CSS module file, no Tailwind.**

Rationale: The entire app uses inline styles. Introducing Tailwind requires: adding `tailwindcss` (v3 or v4), configuring PostCSS, configuring a Vite plugin, and for dark mode with `@tailwindcss/typography` the complexity is significant. For a single markdown viewer component, this is disproportionate overhead.

Instead:

- Create `src/features/artifact-viewer/brief-prose.css` (or `BriefView.module.css` for CSS modules)
- Scope all rules under `.brief-prose { }` to avoid leaking into other components
- Import in `ArtifactViewer.tsx`: `import './brief-prose.css'`
- Vite handles static CSS imports natively; they become a `<link>` tag under `default-src 'self'` CSP

**Key CSS rules to include:**
```css
.brief-prose { color: #e0e0e0; line-height: 1.7; max-width: 80ch; }
.brief-prose h1 { font-size: 1.5rem; border-bottom: 1px solid #333; }
.brief-prose h2 { font-size: 1.25rem; margin-top: 2rem; }
.brief-prose h3 { font-size: 1.1rem; color: #aaa; }
.brief-prose blockquote { border-left: 3px solid #f59e0b; padding-left: 1rem; color: #aaa; }
.brief-prose table { width: 100%; border-collapse: collapse; }
.brief-prose th { background: #1a1a1a; color: #888; font-size: 0.75rem; text-transform: uppercase; }
.brief-prose td, .brief-prose th { padding: 8px 12px; border-bottom: 1px solid #222; }
.brief-prose code { background: #1a1a1a; padding: 2px 6px; border-radius: 3px; font-size: 0.9em; }
.brief-prose pre { background: #111; border: 1px solid #333; border-radius: 8px; overflow: auto; }
.brief-prose hr { border-color: #333; }
```

If Tailwind is desired in the future, this CSS can be mechanically converted to `@tailwindcss/typography` prose overrides.

## 7. Testing Strategy

**Current pattern** (`src/__tests__/App.test.tsx`, `src/__tests__/RunComposer.test.tsx`):
- RTL (`@testing-library/react`) + vitest
- `vi.mock("@tauri-apps/api/core", ...)` to stub all `invoke` calls
- Tests assert text content and DOM structure via `screen.findByText`, `screen.getByRole`
- No snapshot tests currently

**Testing the markdown viewer:**

**Unit tests for BriefView** (add to a new `src/__tests__/ArtifactViewer.test.tsx`):
- Render `<BriefView brief={sampleMarkdown} />` with a known markdown string
- Assert that `screen.getByRole("heading", { name: "Research Brief", level: 1 })` exists
- Assert table rows render: `screen.getByRole("columnheader", { name: /Provider/i })`
- Assert blockquotes render: `document.querySelector("blockquote")` has correct text
- Assert code blocks render syntax-highlighted `<code>` elements

**Integration test for ArtifactViewer**:
- Mock `invoke("get_brief", ...)` to return a realistic brief string (can use one of the Rust test fixtures' expected outputs)
- Verify the Brief tab renders parsed HTML, not raw markdown source

**Accessibility tests:**
- `@testing-library/jest-dom`'s `toHaveAccessibleName()` for headings
- Verify table has `<thead>` and `<tbody>` (react-markdown + remark-gfm produces these)

**What NOT to do:**
- Do not use snapshot tests for rendered markdown HTML. The remark/rehype pipeline output can change between minor plugin versions, making snapshots brittle.
- Do not test highlight.js output classes — these are implementation details of the highlighting library.

## 8. Risk Assessment

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| CSP blocks highlight.js CSS theme | Medium | Low | Import CSS statically via `import 'highlight.js/styles/...'` — Vite bundles it as `<link>`, allowed by `default-src 'self'` |
| react-markdown ESM-only breaks Vitest | Low | Low | Vitest v3 + `environment: 'jsdom'` handles ESM; project already uses ESM (`"type": "module"`) |
| Biome flags `dangerouslySetInnerHTML` | N/A | N/A | Not using it (Option A chosen) |
| Large brief causes render jank | Low | Low | react-markdown renders synchronously but on a separate render pass; React 19's concurrent features can defer; brief content is text, not images |
| remark-gfm parse edge cases in Rust-generated markdown | Low | Low | `render_brief()` output is deterministic and well-formed; no user-supplied markdown; the GFM table format (`|---|---|---|`) matches the remark-gfm parser exactly |
| TypeScript strict mode conflicts with react-markdown types | Low | Medium | react-markdown v10 ships TypeScript types; `noUncheckedIndexedAccess` may require null guards in custom component props — plan for `children ?? null` patterns |
| Tailwind not available for @tailwindcss/typography | N/A | N/A | Chosen approach avoids Tailwind entirely |
| highlight.js bundle size | Low | N/A | Import only needed languages (e.g., `import hljs from 'highlight.js/lib/core'; hljs.registerLanguage('javascript', require(...))`) or use the common bundle |
